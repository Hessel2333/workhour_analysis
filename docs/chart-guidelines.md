# 图表设计规范

本文档用于统一本项目后续新增图表的表达方式，避免不同页面各写一套视觉语言、口径说明和风险提示。

目标不是限制图表数量，而是保证：

- 先讲问题，再讲图形
- 先讲口径，再讲结论
- 先讲边界，再讲联想

## 一、总原则

### 1. 图表先回答问题，不先展示技术

每张图在进入实现前，必须先写清楚它回答的问题。

推荐模板：

- 这张图回答什么问题
- 用户看完后要做什么动作
- 这张图更适合描述、比较、定位还是复盘

如果一句话说不清“这张图到底帮谁做决策”，就不要先画图。

### 2. 优先可解释性，不追求炫技

优先级应始终是：

1. 语义清晰
2. 读图稳定
3. 交互必要
4. 视觉表现

禁止为了“看起来高级”牺牲坐标轴、标签、样本量、来源边界和说明文案。

### 3. 一个图尽量只承担一个主任务

优先做这些单一任务图：

- 排名
- 结构占比
- 趋势变化
- 关系观察
- 异常定位

避免把“趋势 + 结构 + 相关 + 结论”塞进同一张图。

## 二、颜色语义

### 1. 颜色必须先表达语义，再表达美观

当前项目建议统一为以下语义：

- 风险 / 异常 / 返工 / 损耗：红橙系
- 正常 / 健康 / 正向状态：绿色系
- 真实工时主数据：蓝色系
- 规则推导 / derived：青蓝或中性蓝灰
- mock / 示意 / 预留位：橙黄系，并必须配合文字标识
- 中性说明 / 背景分层 / 辅助线：灰蓝系

### 2. 不同语义不要复用同一强调色

例如：

- 不要同时用红色表示“风险”和“重点推荐”
- 不要同时用蓝色表示“真实工时”和“模拟来源”

### 3. 热力图颜色必须符合“越深越强”的直觉

热力图推荐：

- 低值：接近背景的浅色
- 中值：中饱和度色
- 高值：明显更深的同色系

不要使用会让高值和低值都过亮的彩虹色带。

## 三、图表适用场景

### 1. 更适合老板 / 管理者看的图

这类图应该默认可见，并优先放在页面前部：

- 排行条形图
- 100% 堆叠条形图
- 简单趋势图
- 核验漏斗图
- 帕累托图
- 关系详情卡片
- 风险 / 质量 / 结构类摘要图

要求：

- 标题必须是结论导向
- 坐标轴和单位必须完整
- tooltip 之外也要能看懂 70% 信息

### 2. 更适合分析师探索的图

这类图可以保留，但不应抢主视角：

- 完整相关矩阵
- 高维散点图
- 箱线图
- 长尾全量展开图
- 协同分析预留位

要求：

- 默认弱化或折叠
- 说明文字明确“这是探索视图”
- 必须补边界说明

## 四、什么时候必须补充说明元素

### 1. 必须加样本量 n

以下图表必须显示 `n`：

- 相关性详情卡片
- 相关矩阵相关排行
- 任意带回归线或相关强弱判断的图
- 低样本条件下仍保留的统计图

样本量可以放在：

- badge
- subtitle 旁
- 说明区 meta

但不能只藏在 tooltip。

### 2. 必须加阈值线

以下情况应优先加阈值线或参考线：

- 工时高低有明确业务阈值时
- 帕累托图需要强调 80% 覆盖线时
- 风险分或质量分存在明显观察分段时
- 散点图需要给出平均线或中位线帮助读图时

阈值线要用虚线或低对比色，不能压过主数据。

### 3. 必须加中位线或平均线

适用于：

- 散点图
- 箱线图辅助判断
- 需要对比“高于整体还是低于整体”的图

不适用于：

- 本身已经是排行结果的条形图
- 参考线会明显干扰阅读的小样本图

### 4. 必须明确“相关不代表因果”

适用于：

- 所有相关性页面图表
- 任意散点 + 趋势线
- 任意联动分析卡片

这句话不应只放文档里，页面里也要出现。

## 五、坐标轴与标签规范

### 1. 坐标轴名称必须完整

统一规则：

- 百分比：必须带 `%`
- 工时：必须写 `h`
- 日期轴：写 `日期`
- 月轴：写 `月份`
- 周轴：写 `周次`

不要出现“只有刻度、没有语义”的轴。

### 2. 长标签处理规则

对于长项目名、长员工名、长主题名：

- 优先中间省略
- 保留 tooltip 全称
- 不允许因为标签太长把绘图区压到不可读

不要默认让长标签换成多行，把图高无限撑大。

### 3. 分类轴与数值轴要分工明确

- 类别名优先放左侧或下方固定区域
- 数值终点优先留给柱体长度或刻度
- 不要把分类名称跟着柱子终点乱跑

## 六、图表布局规范

### 1. 标题要写成“结论句”，不要只写图种

推荐：

- 单独看返工压力，不和总量混在一张图里
- 总工时高且返工占比高的项目更值得优先复盘

不推荐：

- 项目散点图
- 员工分析图 1

### 2. 图表下方留白必须服务于组件，不是越大越安全

尤其注意：

- x 轴名称
- legend
- visualMap 颜色条

要避免三者互相重叠，也要避免为了保险把热力图底部空出很大一块。

### 3. 热力图和颜色条要当成一个整体布局

要求：

- 图块区域应尽量占满卡片主体
- 颜色条与图块距离适中
- 不允许颜色条变形、竖拉、压到坐标轴标签

### 4. 气泡图必须考虑边界溢出

要求：

- 气泡最大尺寸不能遮挡主要刻度
- 靠近边缘的气泡不能超出绘图区导致裁切异常
- 点过多时应考虑切换范围、分层显示或过滤，而不是强行塞满

## 七、图种选择规则

### 1. 优先使用的图

- 横向条形图：排名、比较、治理优先级
- 100% 堆叠条形图：结构占比
- 折线 / 面积图：趋势变化
- 热力图：密度、分布、切换、日历负载
- 散点图：关系观察
- 漏斗图：流程损耗
- 帕累托图：问题优先级

### 2. 谨慎使用饼图

只有在以下条件同时满足时才可用：

- 分类不超过 3 到 4 类
- 用户只需要看大致占比
- 不需要精确比较相邻类别

否则优先改成条形图或堆叠条形图。

### 3. 禁止使用 3D 图

原因：

- 增加视觉噪声
- 破坏数值判断
- 对工时分析这类管理场景没有额外价值

### 4. 谨慎使用双轴图

仅在两个指标确实需要同时读，而且不容易拆成两张图时才允许使用。  
一旦使用，必须：

- 明确两个轴的单位
- 控制颜色对应关系
- 避免误导用户把两个轴当成同一量纲

## 八、mock / real / derived 展示规则

### 1. 真实图表

可用于：

- 描述现状
- 发现结构问题
- 做复盘入口

仍不可直接用于：

- 个人绩效结论
- 因果判断

### 2. derived 图表

必须明确：

- 来源于规则推导
- 依赖哪些规则或阈值
- 适合作为线索，不是最终结论

### 3. mock 图表

必须同时满足：

- 图表有 mock 标识
- 说明区明确写“仅示意”
- 不进入正式管理结论主链路

## 九、文案规范

每张图优先包含以下四部分：

- `title`：结论导向标题
- `subtitle`：这张图在看什么
- `note`：怎么读
- `caution`：不要误读成什么

推荐语气：

- 这张图回答……
- 当前更适合判断……
- 更适合先看……

避免语气：

- 精准识别
- 真实反映
- 最终判断

## 十、新增图表自检清单

新增任意图表前，至少检查以下问题：

1. 它回答的是描述、比较、趋势、关系还是异常中的哪一种？
2. 标题是否是结论导向，而不是图种名？
3. 坐标轴单位和名称是否完整？
4. 是否需要样本量 `n`？
5. 是否需要阈值线、中位线或平均线？
6. 是否存在 mock / derived 边界说明？
7. 长标签、颜色条、legend、轴名称会不会互相重叠？
8. 是否能在不打开 tooltip 的情况下读懂主要信息？
9. 是否存在更简单的图种可替代？
10. 它会不会诱导用户做超出口径的结论？

## 十一、对当前项目的具体建议

### 1. 首页和项目页

优先保留：

- 排名
- 结构
- 风险优先级
- 趋势

减少：

- 解释成本高但行动价值低的图

### 2. 员工页

优先展示：

- 风险分布
- 救火型识别
- 负载日历热图
- 项目切换热力图

### 3. 相关页

优先展示：

- 候选关系排行
- 关系详情卡片

完整矩阵应作为进阶视图保留。

### 4. 质量页

优先展示：

- 核验漏斗
- 帕累托图
- 质量旗标排行

因为这三类图最接近治理动作，而不是只做展示。
