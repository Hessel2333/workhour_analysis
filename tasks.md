# workhour_analysis 开发任务清单

> 目的：这是给代码 Agent / 开发同学直接执行的任务清单。
> 要求：优先做 P1，再做 P2，最后做 P3。每个任务都应尽量小步提交，避免一次性大改。
> 原则：
> - 不要改变现有产品定位（工时分析 + 数据质量 + 多源扩展）
> - 不要把 mock 数据包装成真实结论
> - 新增指标必须说明口径与限制
> - 图表优先提升“可解释性”，而不是单纯增加炫酷效果

---

## 一、建议文件放置位置

将本文件放在仓库根目录：

- `TASKS.md`

可选补充文件（后续新增）：

- `docs/metric-definitions.md`
- `docs/chart-guidelines.md`
- `docs/data-source-boundary.md`

---

## 二、总体开发优先级

### P1：先做“指标口径稳定化”
目标：让当前分析结果更一致、更可信、更容易维护。

### P2：再做“关键图表增强”
目标：让现有页面更适合老板/管理者阅读。

### P3：最后做“多源协同扩展”
目标：为后续 Git / AI / Feedback 真实接入打基础。

---

# P1 指标口径与结构优化（最高优先级）

## P1-1 新增指标注册层

### 目标
把当前散落在页面里的指标计算逻辑抽离出来，避免重复计算、口径不一致、难以导出。

### 建议新增目录

- `src/lib/metrics/employeeMetrics.ts`
- `src/lib/metrics/projectMetrics.ts`
- `src/lib/metrics/taskMetrics.ts`
- `src/lib/metrics/qualityMetrics.ts`
- `src/lib/metrics/correlationMetrics.ts`
- `src/lib/metrics/index.ts`

### 要求
- 每个文件只负责对应主题指标的纯计算
- 页面层不再直接拼复杂分数
- 每个导出函数都要返回：
  - `value`
  - `label`
  - `description`
  - `formula`（如果是复合指标）
  - `limitations`

### 首批需抽离的指标
- 员工风险分
- 救火指数
- 数据健康分
- 项目返工占比
- 多项目切换率
- 集中度
- 未分类率
- 核验覆盖率

### 验收标准
- `src/pages/*` 中复杂指标计算代码减少
- 页面改为调用统一指标函数
- 任一指标都能追溯公式和口径

---

## P1-2 新增指标口径说明文档

### 目标
避免后续使用者误解“风险分”“救火分”“健康分”等复合分数。

### 新增文件
- `docs/metric-definitions.md`

### 文档至少包含
- 指标名称
- 中文解释
- 计算公式
- 使用字段
- 适用范围
- 不适用场景
- 是否依赖 mock 数据

### 必须覆盖的指标
- 员工风险分
- 救火指数
- 数据健康分
- 项目返工占比
- 项目阶段占比
- AI 深度分（如果保留 mock 展示）

### 验收标准
- 文档可被非开发人员读懂
- 图表 tooltip 或说明区域可引用这份定义

---

## P1-3 参数配置化

### 目标
把当前代码里隐含的阈值提取成可维护参数，方便调口径。

### 建议新增文件
- `src/config/analysisConfig.ts`

### 首批参数
- 日工时超时阈值（默认 8）
- 高强度加班阈值（默认 10）
- 高项目切换阈值（默认 3）
- 低置信度阈值（默认 0.8）
- Top N 默认值（默认 10）
- 返工关键词词典开关
- 阶段识别规则开关
- 是否显示 mock 图表开关

### 要求
- 页面与指标函数统一从配置读取，不写死 magic number
- 可先做静态配置文件，不强求 UI 可编辑

### 验收标准
- 搜索关键阈值时，不再大量散落在页面代码中
- 调整配置后多个页面行为同步变化

---

## P1-4 mock / real 边界增强

### 目标
进一步避免用户把模拟数据误认为真实结论。

### 要求
- 在页面顶部增加全局提示条：
  - 当前真实来源：工时原始数据
  - 当前模拟来源：Git / AI / 用户反馈（如未接真实数据）
- 对依赖 mock 的图表增加统一标识
- 报告导出页也必须保留该说明

### 涉及文件（优先排查）
- `src/pages/OverviewPage.tsx`
- `src/pages/CorrelationPage.tsx`
- `src/pages/ReportPage.tsx`
- `src/components/ChartPanel.tsx`

### 验收标准
- 用户不进入代码也能分辨真实图表与模拟图表
- 导出 PDF 后依然能看到边界说明

---

## P1-5 任务分类可解释性增强

### 目标
让任务分类结果可以被复核，而不是黑盒归类。

### 要求
新增“分类命中详情”能力：
- 显示命中规则名
- 显示命中关键词
- 显示分类置信度
- 显示是否命中 fallback

### 建议新增内容
- `src/lib/topicExplain.ts`
- 在任务页或详情弹窗中展示解释信息

### 可选增强
新增一个自定义修正规则文件：
- `src/config/customTopicRules.ts`

### 验收标准
- 任一任务被分类后，能查看“为什么被这样分类”
- 未分类/低置信度任务可快速复核

# P2 图表与页面增强（高优先级）

## P2-1 项目工作流结构图

### 目标
新增一张最适合老板看的项目比较图。

### 图表建议
**100% 堆叠条形图**

### 展示内容
按项目展示以下工时占比：
- 建设型
- 修补型
- 支撑型
- 成长型

### 数据来源
复用现有任务工作流归并逻辑。

### 建议文件
- `src/pages/ProjectsPage.tsx`
- 或抽成 `src/components/charts/ProjectWorkflowStructureChart.tsx`

### 验收标准
- 一眼能识别哪些项目“修补型占比偏高”
- 默认展示 Top N 项目，可切换全部

---

## P2-2 项目阶段流转图

### 目标
让项目不只看总工时，还能看生命周期阶段演化。

### 图表建议
- 堆叠面积图
- 或按月堆叠柱状图

### 阶段建议
- 需求 / 设计
- 开发
- 联调 / 测试
- 上线 / 发布
- 维护 / 反馈

### 数据来源
复用 `detectTaskStage` 逻辑。

### 涉及文件
- `src/lib/taskSignals.ts`
- `src/pages/ProjectsPage.tsx`

### 验收标准
- 可按项目查看阶段变化
- 至少支持按月聚合

---

## P2-3 核验漏斗图

### 目标
同时展示“数据质量”和“流程完整度”。

### 图表建议
漏斗图或分步条形图

### 建议漏斗层级
- 原始任务数
- 合法任务数
- 已分类任务数
- 已核验任务数
- 高置信度任务数

### 数据来源
复用解析和质量旗标逻辑。

### 涉及文件
- `src/pages/QualityPage.tsx`
- `src/data/workhourData.ts`

### 验收标准
- 能直观看到数据在哪一步损耗最大
- 支持 tooltip 解释每层口径

## P2-4 员工负载日历热图

### 目标
补足“连续高负载”展示，不只看总量排名。

### 图表建议
日历热图 / 矩阵热图

### 维度建议
- 横轴：日期
- 纵轴：员工
- 颜色：工时 / 风险分 / 异常标签数

### 涉及文件
- `src/pages/EmployeesPage.tsx`
- 或 `src/components/charts/EmployeeLoadCalendarChart.tsx`

### 验收标准
- 能快速识别连续高工时员工
- 支持切换显示“总工时 / 风险 / 异常数”

---

## P2-5 相关性详情卡片

### 目标
不要只给相关矩阵，还要给“可讲述”的关系图。

### 图表建议
散点图 + 回归线 + 样本量说明

### 首批建议关系
- 多项目率 vs 集中度
- 修补型占比 vs 总工时
- 返工占比 vs 风险分
- （如保留 mock）AI 深度分 vs 返工率

### 涉及文件
- `src/pages/CorrelationPage.tsx`

### 验收标准
- 每个关系卡片都要显示样本量 n
- 明确显示“相关不代表因果”
- 对弱相关默认弱化展示

---

## P2-6 质量问题帕累托图

### 目标
帮助团队先治理影响最大的少数问题。

### 图表建议
帕累托图（柱 + 累计折线）

### 问题来源
可按质量旗标统计，例如：
- 未核验
- 高项目切换
- 低置信度分类
- 极端工时
- 空任务描述
- 高返工倾向

### 涉及文件
- `src/pages/QualityPage.tsx`

### 验收标准
- 能看出前 20% 问题类型覆盖了多少数据质量风险
- 默认按数量降序

# P3 交互与产品化增强（中优先级）

### 目标
降低学习成本，减少用户自己拼字段筛选。

### 建议快捷入口
- 高返工项目
- 高切换员工
- 低核验覆盖项目

- 分类待复核任务

### 涉及文件
- `src/store/appStore.ts`
- `src/pages/OverviewPage.tsx`
- `src/components/*`

### 验收标准
- 用户点击一个快捷入口即可进入对应视角
- 不要求自己理解所有筛选字段

## P3-2 图表设计规范文档

### 目标
控制后续新增图表的样式和表达方式。

### 新增文件
- `docs/chart-guidelines.md`

- 什么时候必须加阈值线 / 中位线 / 样本量
- 什么情况下禁止使用饼图 / 3D 图

### 验收标准
- 后续新增图表可按规范执行

## P3-3 报告页增强

### 目标

### 涉及文件
- `src/pages/ReportPage.tsx`

### 验收标准

- 报告文字能独立被老板阅读

# P4 多源真实接入准备（后续）

## P4-1 Git 数据接入准备

### 目标
优先接入最容易解释的外部真实数据。

### 为什么先接 Git
相比 AI 与客户反馈，Git 边界更清晰，和工时联动更稳定。

### 先做的事情
- 明确 Git 数据接口格式
- 定义 employee 与 Git identity 的映射方式
- 增加数据导入校验
- 给出空数据 fallback

### 建议新增文件
- `src/data/connectors/gitAdapter.ts`
- `docs/data-source-boundary.md`

### 首批指标
- commit 数
- PR 数
- reviewHours
- LOC 增删

---

## P4-2 AI 与反馈数据边界文档

### 目标
在接真实 AI / Feedback 前先把边界定义清楚。

### 新增文件
- `docs/data-source-boundary.md`

### 文档至少说明
- 哪些字段是模拟的
- 哪些字段未来可接真实接口
- 哪些图表在当前阶段不能用于绩效判断
- 哪些分析只适用于团队层，不适合个人层

### 验收标准
- 新接数据前，团队已明确边界和风险
