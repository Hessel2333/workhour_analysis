# 项目全面审查与优化建议

> 本文基于对项目全部源代码（前端、后端、数据层、分析引擎）的综合审查，定级标准参考 P0（必须修复）/ P1（强烈建议）/ P2（中期演进）。

---

## 一、架构与工程质量

### 🟡 P1 — `analytics.ts` 体量过大，应拆分

**问题**：`src/lib/analytics.ts` 已达 719 行，承担了过滤、聚合、异常检测、相关性计算、报告生成等几乎全部分析逻辑，可维护性差。

**建议**：
- 拆分为 `aggregation.ts`（数据聚合）、`anomaly.ts`（异常检测）、`correlation.ts`（相关性）、`reportBuilder.ts`（报告生成）
- 每个模块单独导出，`analytics.ts` 作为门面（Facade）统一组合调用

---

### 🟡 P1 — 每次 filter 变更触发全量重算，存在性能隐患

**问题**：`patchFilters` 在每次筛选条件变化时都会调用完整的 `buildAnalyticsView()`，该函数会遍历全量数据。在大数据量（如年度数据）下，UI 切换响应会明显变慢。

**建议**：
- 引入 `useMemo` 或 `zustand` 的 selector 做局部缓存
- 对计算密集的子任务（如 Pearson 相关、趋势斜率）加上 debounce 或 `Web Worker` 懒计算

---

### 🟡 P1 — 数据解析在模块加载时同步执行，可能阻塞首屏

**问题**：`workhourData.ts` 第 221 行 `export const defaultDataset = parseWorkhourSource(rawSource)` 在模块加载时**同步**解析 709KB 的 `data.json`，会阻塞 JS 主线程，延缓 React 首次渲染。

**建议**：
- 将 `defaultDataset` 改为通过 `useEffect` + `useState` 异步初始化
- 或使用 `React.lazy` + `Suspense` 将数据加载与首屏渲染解耦

---

### 🟢 P2 — 后端服务器使用裸 `node:http`，缺乏中间件体系

**问题**：`server/index.mjs` 为纯 Node.js 实现，无请求体大小限制、无限流防护、无 rate-limit，API Key 潜在暴露风险（CORS 配置为 `*`）。

**建议**：
- 迁移到 Hono 或 Express.js（Bun/Node 双支持）
- 添加 body size limit（防止超大 prompt 请求）
- 生产环境 CORS 改为白名单域名

---

## 二、数据层与分析引擎

### 🔴 P0 — 连接器数据（Git/AI/Feedback）全部为 Mock

**问题**：`connectors.ts` 中 Git、PR、AI 使用日志、用户反馈均为纯 Mock 生成，分析引擎基于这些假数据产出建议（如 `反馈 mock 平均评分`），可能产生误导性的"分析结论"。

**建议**：
- 在所有使用了 Mock 连接器数据的图表和 AI 建议中，**强制增加"示意数据"徽章**，防止管理者误信
- 制定真实接入路线图（见下方任务清单）

---

### 🟡 P1 — 任务主题分类依赖关键词，语义覆盖弱

**问题**：`topicRules.ts` 使用关键词词典对任务名称分类，复杂任务名（如"前后端联调 / AI 评审接口调整"）容易被错误归类或归为"未分类"，影响主题分析可信度。

**建议**：
- 增加低置信度（`topicConfidence < 0.5`）时的人工复核台
- 可选：在 AgentPage 中集成 Gemini 对"待确认"任务做语义二次分类

---

### 🟡 P1 — 异常检测完全基于固定规则阈值

**问题**：`aggregateEmployeeDays` 的异常评分采用硬编码阈值（如 `reportHour >= 9`、`taskCount >= 3`），不随数据分布自适应，对于个人工作模式差异大的团队误报率高。

**建议**：
- 短期：增加"相对于个人历史均值"的动态阈值（即同一员工的异常定义应个性化）
- 长期：引入 IQR 或 Z-score 等统计方法做基础异常检测

---

### 🟡 P1 — 趋势斜率在低样本下会误导用户

**问题**：`computeTrendSlope` 在样本天数 < 7 时仍会计算并展示斜率，这在统计意义上接近无效，却以正式图表形式呈现。

**建议**：
- 样本天数 < 7 时，趋势斜率改为显示"-"并附"样本不足"提示
- 激活已有的低样本模式展示逻辑以更完整地覆盖此场景

---

## 三、前端与 UI/UX

### 🟡 P1 — 缺少骨架屏（Skeleton Screen）

**问题**：数据上传或重新解析时，页面会在旧内容清空到新内容渲染之间出现空白闪烁。

**建议**：
- 为 `MetricCard`、`ChartPanel`、数据表格组件添加 `loading` propset 骨架屏变体
- 利用 `framer-motion` 实现 fade-in 过渡

---

### 🟡 P1 — 缺少响应式/移动端适配

**问题**：侧边栏固定 232px 宽，`page-grid` 固定 12 列栅格，在小屏幕（< 1024px）下布局严重溢出。

**建议**：
- 侧边栏改为 `@media (max-width: 900px)` 时折叠为 icon-only 或 hamburger menu
- `page-grid` 在小屏改为单列或双列自适应

---

### 🟡 P1 — 全局缺少 Toast/Notification 系统

**问题**：文件解析失败仅在过滤栏显示一行红字，Gemini 分析完成也无任何操作反馈，用户体验割裂。

**建议**：
- 引入轻量 Toast 通知（可自行实现或使用 `sonner`）
- 覆盖场景：文件解析结果（成功/失败）、AI 分析完成、报告导出完成

---

### 🟡 P1 — 图表缺少区域缩放（dataZoom）

**问题**：年维度下图表 X 轴数据点曲线密集，无法放大局部查看，降低了趋势分析的可用性。

**建议**：
- 为趋势图、加班图、员工风险散点图添加 ECharts `dataZoom` 控件
- 至少提供底部滑动条形式（`type: 'slider'`）

---

### 🟢 P2 — 深色模式支持

**问题**：当前 CSS 变量体系已较完善，但未提供 `prefers-color-scheme: dark` 覆盖，在系统深色模式下图表背景与界面产生明显不协调。

**建议**：
- 提取 `@media (prefers-color-scheme: dark)` 变量覆盖
- ECharts 主题同步切换（在 `chartTheme.ts` 中添加 dark 分支）

---

### 🟢 P2 — 报告导出为截图而非正式文档

**问题**：`ReportPage` 的导出功能基于 `html2canvas` + `jspdf`，导出效果接近页面截图，不适合正式管理汇报。

**建议**：
- 设计独立的"报告模板"视图，以结构化排版（标题页、关键指标、图表说明、建议）为目标
- 支持导出预览与模块选择

---

## 四、安全与合规

### 🟡 P1 — Gemini API Key 通过 `.env` 管理，CORS 配置为 `*`

**问题**：本地开发的 CORS 为 `*`，任意来源均可请求 Gemini proxy，若后端部署公开会有 Key 泄漏风险。

**建议**：
- 生产环境（Vercel/Render）的 CORS 白名单化
- 添加请求 IP/来源校验，或通过 Vercel 的 Edge 函数替代裸 Node 服务器

---

### 🟢 P2 — 缺少敏感文本脱敏与合规说明

**问题**：员工姓名在大多数图表中以真实姓名展示，AI Prompt 中也包含员工姓名，缺少明确的隐私边界。

**建议**：
- 增加全局"脱敏模式"开关，切换后所有员工姓名改用 `maskedLabel`
- 在方法说明页增加隐私与合规声明
