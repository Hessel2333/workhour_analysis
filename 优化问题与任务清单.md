# 工时分析项目优化问题与任务清单

## 背景
- 本文基于当前已实现的本地网页应用、根目录研究文档 [deep-research-report_ChatGPT.md](/Users/tian/Codes/workhour_analysis/deep-research-report_ChatGPT.md) 与 [deep-research-report_Gemini.md](/Users/tian/Codes/workhour_analysis/deep-research-report_Gemini.md) 整理。
- 目标是提升数据可信度、可解释性、管理可读性，并为真实多源协同接入打基础。

## 主要问题

### 1. 数据基础与口径
- Git、PR、用户反馈、AI 使用日志仍以 mock 为主，真实协同分析尚未落地。
- 缺少统一数据字典，老板无法判断“异常员工日”“趋势斜率”“集中度”等指标的计算口径。
- 还没有形成显式的身份映射、项目映射、时间粒度对齐方案。
- 数据质量只有 flag，没有统一质量分、覆盖率影响范围、图表受影响说明。

### 2. 分析方法与结论可信度
- 当前异常检测主要是规则阈值，缺少统计异常、分层模型、滞后分析和显著性判断。
- 相关性图表缺少样本量、显著性、风险提示，容易被误解为因果。
- 短样本下仍显示趋势斜率和分布判断，缺少低样本降级机制。
- 任务主题分类主要基于关键词，复杂任务语义容易被错误归类。

### 3. 可解释性与交互
- 图表大多只有一句 note，没有“来源 / 方法 / 可靠性 / 注意事项”四段式说明。
- 用户难以区分真实数据、示意数据、规则推导数据、模型生成文本。
- 缺少从 KPI 到图表到明细到原始任务的稳定下钻路径。
- 智能体输出虽有建议，但缺少回链证据图、图表定位与复盘入口。

### 4. UI 与视觉表达
- 颜色好看但语义弱，风险、可信度、数据来源没有统一视觉编码。
- 示例数据虽已加 badge，但仍缺少全局来源提示和更强的页面级说明。
- 低样本、低可信、规则推导等高风险状态没有形成统一的提醒样式。
- 报告导出更像页面截图，而不是正式的管理层报告模板。

### 5. 模型与治理
- Gemini 已打通，但缺少 prompt 版本管理、输出结构约束、失败降级和调用审计。
- AI 分析结果仍以自由文本为主，不利于后续归档和二次分析。
- 合规治理尚未形成系统提示，如敏感文本脱敏范围、匿名/实名边界、使用目的声明。

## 优化任务清单

### P0 本轮优先
- 建立统一的数据来源标签系统。
- 为核心图表增加来源、方法、可靠性、注意事项说明。
- 增加全局数据健康分和低样本提示。
- 在相关性页面强化“相关不等于因果”和短样本限制。
- 在质量页面补充质量分、覆盖率和报告可靠性说明。
- 将问题与任务清单固化到仓库文档，便于后续持续迭代。

### P1 下一阶段
- 落地真实 Git / PR / AI / 用户反馈接入。
- 建立员工身份映射、项目映射和时间对齐表。
- 为任务分类增加人工词典维护与低置信度复核台。
- 为智能体输出增加结构化 JSON、证据权重和异常等级。
- 建立老板视角与项目经理视角两套工作台。
- 重做报告导出模板，支持正式汇报版。

### P2 中期演进
- 引入 Spearman、显著性估计、分层模型和滞后相关分析。
- 加入聚类画像、因果假设图和质量领先指标体系。
- 建立模型调用缓存、审计日志和失败降级策略。
- 建立数据治理页，包含合规说明、口径说明、版本说明。
